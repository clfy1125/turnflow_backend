Q. "DM이 보내졌음"은 언제 확정되나?

2단계입니다. ① ACCEPTED = Meta가 200 + message_id + recipient_id 응답(제출 성공). ② DELIVERED = echo 웹훅 또는 Conversations API 능동 조회로 실제 도착 확인. 사용자에게 "도착함"이라 보고 가능한 건 DELIVERED 이상입니다.

Q. 실패하면 어떻게 되나? (v3.9)

일시 오류(rate-limit/5xx/transient)는 횟수 상한으로 죽지 않고 무한 defer(QUEUED+next_retry_at) → requeue_deferred_dms가 FIFO 재투입. 유일한 graceful 종결 = 메시징 윈도우 만료(comment 7일/user 24h) → FAILED_WINDOW. token/param은 즉시 종결. 200 받았는데 35분 도착 미확인이면 FAILED_NO_TRACE인데, 이건 '실패'가 아니라 '미확인'(total_unconfirmed)으로 분리 집계해 성공률을 깎지 않습니다.

Q. 시간당 한도를 넘으면 드랍되나?

아니요. 드랍이 아니라 지연입니다. 계정당 Meta 안전속도(rate_governor 750/hr Private Reply + 분당 버스트)나 캠페인 시간당 한도를 넘으면 다음 윈도우까지 defer 후 순차 재발송됩니다. 댓글 웹훅이 유실돼도 poll_missed_comments(매시간)가 1시간 내 보정 발송합니다.

Q. 토큰이 만료돼 실패한 DM은 영영 못 보내나? (v3.10)

아닙니다. FAILED_TOKEN/SKIPPED로 종결된 건은 제자리 되살림(revive)으로 같은 row·같은 idempotency_key를 QUEUED로 되돌려 재발송합니다(중복 INSERT 불가). 토큰 갱신이 성공하면 그 계정의 실패 건이 자동 되살림되고, 프리미엄 사용자는 retry-failed API로 수동 복구할 수 있습니다. 과발송으로 인스타가 계정을 차단(Action Block, code 368)하면 24h→×2 에스컬레이팅 쿨다운 동안 그 계정 발송을 멈췄다가 자동 재개합니다(차단 연장 방지). 자세한 내용은 §11.

"확정"은 2단계입니다

1단계 · ACCEPTED

접수 확정 — "Meta가 받았다"

  • 조건: HTTP 200 + message_id + recipient_id
  • mark_accepted() → meta_message_id·accepted_at 저장
  • ⚠️ "제출 성공"일 뿐 도착을 의미하지 않음 → 한 단계 더 검증
2단계 · DELIVERED

도착 확정 — "실제로 도착했다"

  • 1차(push): echo 웹훅 is_echo:true mid 매칭
  • 2차(pull, 안전망): GET /{message_id} 조회 성공
  • 두 경로 모두 오면 verified_via=both. 사용자에게 "도착함" 보고 가능 = DELIVERED 이상
💡 99.9% 보증의 핵심 아이디어 웹훅(push)은 유실될 수 있다고 가정하고, 능동 조회(pull) + 주기 워커로 모든 ACCEPTED 건을 교차검증합니다. 35분간 어느 신호도 없으면 FAILED_NO_TRACE로 종결해 "떠 있는 상태"를 남기지 않습니다.

재시도 백오프 — v3.9: 무한 defer (횟수 상한 없음)

1회60s→ 2회120s→ …2배씩→ n회최대 1h 고정→ 메시징 윈도우 만료FAILED_WINDOW

산식 min(60 × 2^min(retry_count,10), 3600) — 상한 1h(쿼터는 시각 경계로 풀림). retriable은 5회 후 죽지 않는다; QUEUED로 회귀 후 requeue_deferred_dms가 FIFO 재투입하며, 종료는 오직 윈도우 만료(send_dm_task 진입부 age 가드)로만 일어납니다. (parent_log 있는 reward/retry child는 sent/failed/unconfirmed 모든 카운트 제외)

🆕 code 1/2 "error-but-delivered" 재발송 전 검증 — v3.10.1 (P12) except DMSendError 블록(tasks.py:947)이 maybe_delivered = isinstance(e, DMAnomalyError) or e.code in (1, 2)로 anomaly(§8.2)와 code 1/2를 동일 취급 — 재발송 전 has_recent_message_to_recipient(since_seconds=900)로 이미 보냈는지 확인합니다.
  • recent=True → mark_accepted("") + increment_sent(opening만) + mark_delivered(CONV_API), 재발송 안 함(오프닝 DM 2개 중복 방지).
  • recent=None(조회가 타임아웃/5xx로 불확실) → code 1/2는 defer+retry(무손실; reconcile_stuck_submitting의 None→requeue와 대칭). ⚠️ 비대칭: anomaly(200-no-mid)만 None→FAILED_NO_TRACE로 종결(재발송 안 함), code 1/2를 여기서 종결하면 유실 위험이라 의도적.
  • recent=False → 기존 defer 재발송.
  • 명시적 rate-limit {4,17,32,368,613}은 이 검증을 건너뜀 — 요청 거부라 전달이 없어 확인 불필요.
  • ⚠️ 잔여 한계: 전달됐는데 Conversations 인덱싱 지연으로 recent=False면 재시도 중복 가능(기존과 동일 = 회귀 아님).

7 관측 / 통계

도착률(delivery_rate)

delivery_rate = (delivered + read) / (accepted + delivered + read + failed_no_trace)

분모는 "Meta가 접수한 이후" 결과만 셉니다(큐/스킵 제외). 소수 4자리 반올림. _build_stats·compute_campaign_enrichment·dashboard 모두 동일 공식이며 분모 0이면 0.0. 단 recent_delivery_rate_24h는 분모 0이면 null 반환.

⚠️ 사용량 카운트 주의 billing.UsageCounter.dm_sent는 stale(증가 안 됨). 월 DM 사용량은 SentDMLog로 직접 집계하고 한도는 PlanLimits 기준입니다.
🆕 도착 미확인(unconfirmed) 분리 — v3.9 total_unconfirmed(migration 0023)는 FAILED_NO_TRACE 전용 카운터입니다. "보낸 게 실패"가 아니라 "도착을 확인 못 함"이므로 total_failed·success_rate에서 분리 집계해 성공률을 부당하게 깎지 않습니다. child(reward/retry)는 sent/failed/unconfirmed 모두 제외.

부분 해결 8.1b rate_governor 고정 윈도우 카운터는 발송 시도마다 소비됨

  • rate_governor.check()는 호출 즉시 INCR — defer 재시도 시 같은 1건이 여러 번 소비될 수 있음(보수적 과소발송, 밴 위험↓).

  • v3.10.1 — Redis 유실 즉시 복구: fail-closed 의 최대 1h 동결이 유일 회복이 아닙니다. rate_governor.rehydrate_from_db()가 SentDMLog.submitted_at 윈도우에서 계정별 시/분 카운터를 재구성 AND dmrate:reset_until을 삭제(동결 즉시 해제) → dr_catchup STEP0 + 활성 사이트 Celery worker_ready에서 호출. Redis 유실 후 1h 멈추지 않고 재수화하여 즉시 재개합니다.

해결됨 ✅ 8.2 200-no-msgid anomaly + code 1/2 재발송 중복 위험 → 검증 후 재발송

  • (과거) anomaly 가 rate_limited로 분류돼 그냥 재발송 → Meta 가 이미 보냈으면 중복 DM.
