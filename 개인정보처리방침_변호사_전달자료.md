## 1. 서비스 개요 및 개인정보 처리 구조

### 1.1 서비스 설명

TurnFlow는 **AI 기반 링크인바이오(Link-in-Bio) 페이지 빌더** 및 **인스타그램 비즈니스 계정 자동화** SaaS 서비스입니다. 서비스 내에서 개인정보가 처리되는 주요 영역은 다음과 같습니다.

### 1.2 개인정보 처리가 발생하는 영역

| 영역 | 정보주체 | 설명 |
|------|----------|------|
| **회원 가입·인증** | 회원 | 이메일 기반 가입, JWT 인증 |
| **구독·결제** | 유료 회원 | PayApp PG사를 통한 월간 정기결제 |
| **인스타그램 연동** | 회원 (비즈니스 계정 보유자) | Meta OAuth를 통한 인스타그램 계정 연결 |
| **링크인바이오 페이지** | 회원 (페이지 제작자) | 페이지 콘텐츠 제작·관리 |
| **페이지 방문자 추적** | 비회원 방문자 | 페이지 조회수·클릭 수 통계 수집 |
| **문의·구독 폼** | 비회원 방문자 | 크리에이터 페이지 내 문의 제출·뉴스레터 구독 |
| **AI 콘텐츠 생성** | 회원 | LLM 기반 바이오·테마·카피 생성 |
| **워크스페이스·팀** | 회원 | 다중 사용자 협업 공간 |

### 1.3 정보주체 구분

| 정보주체 | 정의 |
|----------|------|
| **회원** | TurnFlow에 이메일로 가입한 서비스 이용자 |
| **워크스페이스 멤버** | 초대를 통해 워크스페이스에 참여한 회원 |
| **페이지 방문자** | 크리에이터가 만든 링크인바이오 페이지를 열람하는 비회원 |
| **문의 제출자** | 크리에이터 페이지 내 문의 폼에 정보를 입력한 방문자 |

## 2. 수집하는 개인정보 항목 및 수집 목적

### 2.1 필수 수집 정보 (회원가입)

| 수집 항목 | 데이터 형식 | 수집 목적 | 수집 방법 |
|-----------|------------|-----------|-----------|
| **이메일 주소** | EmailField (고유값) | 회원 식별, 로그인 ID로 사용, 서비스 관련 통지 | 회원가입 시 직접 입력 |
| **비밀번호** | CharField (해시 처리) | 본인 인증, 계정 보안 | 회원가입 시 직접 입력 |
| **이름(성명)** | CharField (선택) | 서비스 내 표시명, 회원 식별 보조 | 회원가입 시 직접 입력 |

- **비밀번호 저장 방식**: PBKDF2 또는 Bcrypt 단방향 해시 처리 (원문 저장하지 않음)
- **이메일**: 로그인 ID(USERNAME_FIELD)로 사용됨

### 2.2 서비스 이용 과정에서 수집되는 정보 (회원)

| 수집 항목 | 수집 시점 | 수집 목적 |
|-----------|----------|-----------|
| **가입 일시** (`date_joined`) | 회원가입 | 서비스 이용 기간 관리, 통계 |
| **최종 로그인 일시** (`last_login`) | 로그인 시마다 | 계정 활동 모니터링, 보안 |
| **구독 요금제 정보** (플랜명, 상태, 시작일, 종료일) | 요금제 변경 시 | 서비스 이용 권한 관리 |
| **유료 전환 일시** (`pro_activated_at`) | 유료 결제 시 | 환불 기준일 산정 (7일 이내 환불) |
| **AI 토큰 잔액 및 사용 이력** | AI 기능 사용 시 | 이용량 정산, 과금 관리 |
| **워크스페이스 정보** (이름, 역할, 멤버 목록) | 워크스페이스 생성·참여 시 | 팀 협업 기능 제공 |
| **워크스페이스 초대 이메일** | 멤버 초대 시 | 초대 수신자 식별 |

### 2.3 인스타그램 연동 시 수집 정보

| 수집 항목 | 수집 방법 | 수집 목적 |
|-----------|----------|-----------|
| **인스타그램 사용자명** (`username`) | Meta OAuth 인증 후 API 조회 | 연결 계정 식별·표시 |
| **인스타그램 계정 ID** (`external_account_id`) | Meta OAuth 인증 후 API 조회 | 계정 고유 식별, API 호출 |
| **계정 유형** (Business/Creator) | Meta API 조회 | 이용 가능 기능 판별 |
| **OAuth 액세스 토큰** | Meta OAuth 인증 시 발급 | Instagram API 호출을 위한 인증 |
| **토큰 만료일** | Meta API 응답 | 토큰 갱신 관리 |
| **부여된 권한(Scopes)** | OAuth 동의 시 확인 | 이용 가능 API 범위 관리 |

- **OAuth 시 요청하는 권한(Scopes)**: 페이지 목록 조회(`pages_show_list`), 페이지 참여도 조회(`pages_read_engagement`), 기본 프로필/미디어 조회(`instagram_basic`), 댓글 관리(`instagram_manage_comments`), DM 관리(`instagram_manage_messages`), 비즈니스 관리(`business_management`)

### 2.4 링크인바이오 페이지 관련 정보

| 수집 항목 | 수집 목적 |

| **커스텀 CSS** | Pro+ 회원 페이지 스타일 적용 |

> - 현재 수신 거부(Unsubscribe) 메커니즘이 미구현 상태이므로, 구현 의무 여부 검토가 필요합니다.

## 3. 자동 수집 정보

### 3.1 페이지 방문자 추적 (PageView)

크리에이터가 만든 링크인바이오 페이지를 방문하는 모든 사용자(회원·비회원)로부터 다음 정보가 자동 수집됩니다.

| 수집 항목 | 처리 방식 | 저장 형태 | 수집 목적 |
|-----------|----------|-----------|-----------|
| **IP 주소** | SHA-256 해시 처리 | 64자 해시 문자열 (원본 IP 미저장) | 방문자 중복 판별 (Unique Visitor 산출) |
| **국가 코드** | Cloudflare 헤더(`CF-IPCountry`) 또는 GeoIP2 DB 조회 | ISO 3166-1 2글자 코드 (예: KR, US) | 방문자 국가별 통계 |
| **유입 경로 (Referer)** | HTTP Referer 헤더에서 도메인 추출 | 도메인명 (예: "Instagram", "Google") | 유입 채널 분석 |
| **방문 일시** | 서버 시간 자동 기록 | DateTimeField | 시간대별 통계 |

### 3.2 블록 클릭 추적 (BlockClick)

| 수집 항목 | 처리 방식 | 저장 형태 | 수집 목적 |
|-----------|----------|-----------|-----------|
| **IP 주소** | SHA-256 해시 처리 | 64자 해시 문자열 (원본 미저장) | 클릭 중복 판별 |
| **국가 코드** | Cloudflare 헤더 또는 GeoIP2 | 2글자 코드 | 국가별 클릭 통계 |
| **유입 경로** | HTTP Referer 도메인 추출 | 도메인명 | 유입 채널별 클릭 분석 |
| **클릭 일시** | 서버 시간 자동 기록 | DateTimeField | 시간대별 통계 |
| **클릭한 블록/링크 ID** | 프론트엔드 이벤트 전달 | 블록 ID, 링크 ID | 블록별 성과 분석 |

### 3.3 IP 주소 처리 상세

```
[처리 흐름]
1. HTTP 요청 수신 → X-Forwarded-For 헤더 또는 REMOTE_ADDR에서 IP 추출
2. IP 주소를 SHA-256 해시 함수로 단방향 변환
3. 해시 값만 DB에 저장 (원본 IP 즉시 폐기)
4. 해시로부터 원본 IP 복원은 기술적으로 불가
```

### 3.4 국가 판별 상세

```
[판별 우선순위]
1순위: Cloudflare CF-IPCountry 헤더 (CDN 레벨)
2순위: nginx X-Country-Code 헤더 (웹서버 레벨)
3순위: GeoIP2 로컬 DB 조회 (MaxMind GeoLite2-Country.mmdb)
→ 결과: ISO 3166-1 alpha-2 국가 코드 (예: KR, US, JP)
```

### 3.5 유입 경로(Referer) 처리 상세

```
[처리 흐름]
1. HTTP Referer 헤더에서 전체 URL 수신
2. URL에서 도메인명만 추출
3. 도메인을 소스명으로 매핑 (예: instagram.com → "Instagram", google.com → "Google")
4. 매핑된 소스명을 DB에 저장
→ 매핑 목록: Instagram, Facebook, Google, Naver, YouTube, Twitter/X, TikTok, 
   LinkedIn, Pinterest, KakaoTalk, Reddit, Threads 등 30여개 플랫폼
```

> **법적 검토 필요**: 
> - IP 해시 처리가 개인정보보호법상 '비식별 처리'로 인정받을 수 있는지 확인 필요
> - 비회원 방문자에 대한 자동 수집의 법적 근거(정당한 이익 등) 검토 필요
> - 페이지에 방문자 추적 안내 문구(쿠키 배너 등) 표시 의무 여부 검토 필요

> - **AI 생성물의 정확성 면책**: AI 생성 콘텐츠의 허위·부정확 정보에 대한 책임 범위

## 9. 개인정보 보호 기술적·관리적 조치

### 9.1 암호화 조치

| 대상 | 암호화 방식 | 상세 |
|------|------------|------|
| **비밀번호** | PBKDF2 / Bcrypt 단방향 해시 | Django 기본 `make_password()` 사용, 원문 복원 불가 |
| **인스타그램 액세스 토큰** | Fernet 대칭키 암호화 (AES-128-CBC + HMAC) | Django SECRET_KEY 기반 키 파생 (SHA-256), DB 저장 시 암호화·조회 시 자동 복호화 |
| **IP 주소** | SHA-256 단방향 해시 | 원문 복원 불가, 비식별화 목적 |
| **JWT 토큰** | HS256 (HMAC-SHA256) 서명 | SECRET_KEY를 서명 키로 사용 |

### 9.2 접근 통제

| 조치 | 상세 |
|------|------|
| **인증** | JWT 기반 인증 (액세스 1시간, 리프레시 7일) |
| **권한 관리** | 역할 기반 접근 제어(RBAC): Owner / Admin / Member |
| **데이터 격리** | 워크스페이스 단위 논리적 데이터 분리 |
| **API 인증** | 모든 API 호출 시 Bearer 토큰 필수 |
| **관리자 계정** | Django Admin 별도 인증 (`is_staff` / `is_superuser`) |

### 9.3 전송 구간 보안

| 조치 | 상세 |
|------|------|
| **HTTPS** | 운영 환경 전체 HTTPS 적용 (Caddy 리버스 프록시) |
| **HSTS** | Caddy를 통한 HTTP Strict Transport Security 적용 |
| **CSRF 보호** | Django CSRF 미들웨어 활성화, 신뢰 Origin 목록 관리 |
| **CORS** | 허용된 Origin만 교차 출처 요청 허용 (`CORS_ALLOWED_ORIGINS`) |
| **보안 헤더** | `X-Content-Type-Options: nosniff`, `X-XSS-Protection: 1; mode=block`, `X-Frame-Options: DENY` |

### 9.4 웹훅 보안

| 조치 | 상세 |
|------|------|
| **인증 키 검증** | PayApp 웹훅 수신 시 `userid`, `linkkey`, `linkval` 일치 확인 |
| **멱등성 처리** | 거래번호(`mul_no`) + 결제상태(`pay_state`) 조합 유니크 제약조건으로 중복 방지 |
| **재시도 처리** | PayApp 측 최대 10회 재시도 (응답 실패 시) |

### 9.5 인프라 보안

| 조치 | 상세 |
|------|------|
| **컨테이너화** | Docker 기반 격리 운영 |
| **환경변수 관리** | 민감 정보(SECRET_KEY, DB 비밀번호, API 키 등)는 환경변수로 관리, 코드에 미포함 |
| **요청 추적** | 모든 요청에 UUID 기반 Request ID 부여 (`X-Request-ID` 헤더) |

## 10. 쿠키 및 세션 관리

### 10.1 쿠키 사용 현황

| 쿠키 유형 | 설명 | 유효기간 | 설정값 |
|-----------|------|----------|--------|
| **세션 쿠키** (`sessionid`) | OAuth 팝업 플로우 시 세션 유지 목적 | 1시간 (`SESSION_COOKIE_AGE=3600`) | `HttpOnly=True`, `Secure=True`(운영), `SameSite=None` |
| **CSRF 쿠키** (`csrftoken`) | CSRF 공격 방어 | 세션 종료 시 | `Secure=True`(운영) |

### 10.2 쿠키 보안 설정

| 설정 | 개발 환경 | 운영 환경 |
|------|----------|-----------|
| `SESSION_COOKIE_HTTPONLY` | True | True |
| `SESSION_COOKIE_SECURE` | False | True |
| `CSRF_COOKIE_SECURE` | False | True |
| `SESSION_COOKIE_SAMESITE` | None | None |
| `SESSION_COOKIE_AGE` | 3600초 (1시간) | 3600초 (1시간) |

### 10.3 JWT 토큰 저장

| 항목 | 설명 |
|------|------|
| 저장 방식 | 클라이언트 측 (localStorage 또는 httpOnly 쿠키, 프론트엔드 구현에 따라 상이) |
| 액세스 토큰 유효기간 | 1시간 |
| 리프레시 토큰 유효기간 | 7일 |
| 토큰 갱신 | 리프레시 토큰 사용 시 자동 교체(Rotation), 이전 토큰 블랙리스트 등록 |

> **법적 검토 필요**: 쿠키 사용에 대한 동의 절차(쿠키 배너) 필요 여부, 정보통신망법 관련 쿠키 고지 의무 확인

## 14. 법적 검토 요청 사항 종합

아래는 개인정보처리방침 작성 시 변호사님의 특별한 검토가 필요한 사항을 종합 정리한 것입니다.

### 14.1 수집·이용 관련

- [ ] 필수 수집 항목과 선택 수집 항목의 적정성 검토
- [ ] 비회원 방문자 자동 수집(IP 해시, 국가 코드, Referer)의 법적 근거 (동의 vs. 정당한 이익)
- [ ] 문의 폼·뉴스레터의 개인정보 수집 주체 판단 (TurnFlow vs. 크리에이터)
- [ ] 인스타그램 연동 시 수집되는 제3자(댓글 작성자) 데이터 처리의 법적 근거
- [ ] IP 해시(SHA-256) 처리가 개인정보보호법상 '비식별 처리'로 인정되는지 여부

### 14.2 보유·파기 관련

- [ ] 회원 탈퇴 시 즉시 파기 vs. 유예기간(재가입 방지 등) 설정의 적법성
- [ ] 법정 의무 보관 대상 데이터의 분리 보관 기준 및 방법
- [ ] 방문자 통계, AI 이력 등 보유 기간 설정 기준 자문
- [ ] 로그 파일 내 개인정보(이메일)의 보관 기간 적정성

### 14.3 위탁·제3자 제공 관련

- [ ] PayApp과의 개인정보 처리 위탁 계약 내용 검토
- [ ] Meta(Facebook/Instagram)로의 개인정보 **국외 이전** 동의 요건 및 고지 방법
- [ ] PASS/KCP/NICE 본인인증 도입 시 개인정보 처리 위탁 관계 구성
- [ ] 클라우드 서버 제공업체와의 위탁 계약 필요 여부 및 내용
- [ ] LLM 서비스 경유 시 데이터 제3자 제공 해당 여부 (자체 운영 vs. 외부 모델 사용)

### 14.4 AI 관련 (개인정보보호위원회 가이드라인)

- [ ] AI 학습에 사용자 데이터를 사용하지 않는다는 명시의 법적 유효성
- [ ] 사용자 자유 입력 텍스트에 포함될 수 있는 개인정보의 처리 방안
- [ ] 댓글 자동 분류·숨김 처리가 **자동화된 의사결정**에 해당하는지 판단
- [ ] 자동 DM 발송이 정보주체(댓글 작성자)의 권리에 미치는 영향 검토
- [ ] AI 자동화 처리에 대한 거부권·설명요구권 보장 범위 및 방법

### 14.5 정보주체 권리 관련

- [ ] 정보주체 권리 행사 절차 (접수 채널, 본인 확인, 처리 기한) 법정 요건 확인
- [ ] 데이터 이동권(Data Portability) 의무 적용 여부
- [ ] 미성년자(14세 미만) 가입 시 법정대리인 동의 절차 필요 여부
- [ ] 비회원 방문자의 정보주체 권리 행사 방법 (본인 확인 불가 상황)

### 14.6 기타

- [ ] 개인정보 보호책임자(CPO) 지정 법적 의무 요건 (사업 규모 기준)
- [ ] 쿠키 동의 절차(쿠키 배너) 국내법상 의무 여부
- [ ] 개인정보처리방침의 공개 방법 및 시행일 관리 요건
- [ ] 개인정보 유출 시 통지·신고 의무 절차 (72시간 이내 등)
- [ ] 해외 이용자 대상 서비스 확장 시 GDPR/CCPA 등 해외 법령 적용 가능성

```
[회원가입]
  사용자 → TurnFlow 서버 (이메일, 비밀번호, 이름)
           → PostgreSQL DB (비밀번호는 해시 처리 후 저장)

[인스타그램 연동]
  사용자 → Meta OAuth 페이지 (브라우저)
         → Meta 서버 (인증 코드 발급)
         → TurnFlow 서버 (인증 코드 → 액세스 토큰 교환)
         → PostgreSQL DB (토큰 Fernet 암호화 저장, 사용자명·계정ID 저장)

[결제]
  사용자 → TurnFlow 서버 (구독 요청)
         → PayApp API (상품명, 금액, 결제주기)
         → PayApp 결제 페이지 (사용자가 카드/계좌 직접 입력, TurnFlow 미경유)
         → PayApp → TurnFlow 웹훅 (거래번호, 결제상태, 영수증URL)
         → PostgreSQL DB (거래 기록 저장)

[페이지 방문자 추적]
  방문자 → TurnFlow 서버 (HTTP 요청)
         → 미들웨어 (IP → SHA-256 해시, Cloudflare 헤더 → 국가코드, Referer → 도메인 추출)
         → PostgreSQL DB (해시IP, 국가코드, 도메인명, 타임스탬프 저장)

[AI 콘텐츠 생성]
  회원 → TurnFlow 서버 (컨셉, 스타일, 기존 블록 데이터)
       → Celery 워커 (프롬프트 조립)
       → LLM 서비스 (llm.clfy.ai.kr) (시스템 프롬프트 + 사용자 입력)
       → TurnFlow 서버 (AI 생성 결과)
       → PostgreSQL DB (입력·프롬프트·결과 모두 저장)

[문의 폼 제출]
  방문자 → TurnFlow 서버 (이름, 이메일, 전화번호, 문의 내용, 동의 여부)
         → PostgreSQL DB (문의 데이터 저장)
         → 크리에이터(페이지 운영자)가 관리자 패널에서 조회

[자동 DM 발송]
  인스타그램 댓글 이벤트 → Meta 웹훅 → TurnFlow 서버
         → Celery 워커 (댓글 분류: 관심/스팸/악성/리드)
         → Meta Graph API (자동 DM 발송)
```
